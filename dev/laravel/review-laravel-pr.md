# Laravel pull request review instructions

Provide a code review for the given pull request.

## Non-negotiable rules
- Never change the PR title or its description. Do not edit the PR in any way.

## Core assumptions

- All available tools are functional. Do not test tools or make exploratory calls.
- Only call a tool when it is required to complete the task.
- Act as a single reviewer. Do not rely on subagents, parallel agents, model-specific roles, or multi-agent workflows.
- Prefer high-signal findings only. False positives are worse than silence.
- When feedback relates to a specific piece of code, leave an inline PR review comment for that code.

## Review goal

Find only issues that are highly likely to be real and useful to the author.

Flag issues only when at least one of the following is true:

- The code will fail to compile, parse, or run due to a clear syntax, type, import, or reference problem.
- The code will definitely produce incorrect behavior based on the changed code itself.
- The change introduces a clear security problem that is evident from the changed code itself.
- The code is not performant enough, e.g. N+1 database queries in a loop, or the change degrades performance.

Do not flag:

- Style preferences
- General code quality concerns
- Speculative concerns that depend on unknown runtime state or inputs
- Pedantic nitpicks
- Issues that are likely to be caught by linters or formatters
- Pre-existing issues not introduced by the pull request
- Anything you cannot validate with high confidence

## Required workflow

1. Create a short internal todo list before starting.

2. Check whether review should be skipped. Stop if any of the following is true:
   - The pull request is closed.
   - The pull request clearly does not need code review, such as an automated PR or a trivial change that is obviously correct.

   Notes:
   - Still review pull requests generated by AI or automation if they otherwise need review.
   - To determine whether the agent has already reviewed the PR, inspect the PR comments and review comments using the available GitHub tooling.

3. Read the pull request title and description for author intent.

4. Read the pull request diff and produce a brief working summary of the change for yourself.

5. Review the change from these perspectives, sequentially in a single-agent flow:
   - Clear bugs in the changed code
   - Security problems that are evident from the changed code itself
   - Logic errors that are evident from the changed code itself
   - Performance issues that are evident from the changed code itself

   Keep the review limited to the changed code unless extra local context is required to validate a suspected issue.

6. Validate every candidate issue before reporting it.
   For each possible issue, confirm all of the following:
   - The issue is introduced or exposed by this pull request.
   - The issue is real with high confidence.
   - The issue matters enough to comment on.

   If any of those checks fail, discard the issue.

7. Deduplicate issues so each unique problem is reported once.

8. Produce the review result:
   - If no issues remain, output you are done and how confident you are that merging the PR will not cause problems:

     `No issues found. Checked for bugs, security issues, clear logic problems, and performance. I am 90% confident that this PR is safe to merge.`

   - If you are less then 90% confident that the PR is safe to merge, but found no clear issues, output a summary of your concerns:

     `No clear issues found, but I have some concerns about this PR. I am only 70% confident that it is safe to merge because of [reasons].`

   - If issues remain, output a concise summary listing each issue.

9. Post review comments using these rules:
   - If an issue relates to a specific line or block of code, post it as an inline PR review comment.
   - If a fix is small and self-contained, include a committable suggestion block only when applying that suggestion would fully resolve the issue.
   - If a fix is broader, spans multiple locations, or needs follow-up work, explain the issue and the direction of the fix without a suggestion block.
   - Post only one comment per unique issue.
   - Do not post a separate summary comment when the findings are code-specific and already covered by inline comments.
   - When posting comments with code parts, e.g. function names, variables, etc, always surround them with backticks for clarity.

10. When writing inline comments:
   - State the problem clearly and briefly.
   - Explain why it is a real issue.
   - Include a stable code link when referring to another file.
   - When linking to repository files, use a full commit SHA and a line range with at least one line of surrounding context.

## Commenting policy

Use inline PR review comments by default whenever the feedback relates to a piece of code.

Only avoid inline comments when there is no specific code location that reasonably anchors the feedback. In that case, provide the finding in the review summary instead of forcing a misplaced inline comment.

## Guidance for high-signal reviews

A strong finding usually has all of these characteristics:

- It is specific.
- It is actionable.
- It is grounded in the diff.
- It is likely to affect correctness or safety or performance.
- It can be explained in a few sentences.

Examples of strong findings:

- A newly introduced unresolved symbol or missing import
- A branch that returns the wrong value regardless of input
- A missing null check that makes the new code definitely crash on an already-reachable path
- A newly introduced authorization, validation, or data exposure issue that is directly visible in the diff
- A newly introduced N+1 database query in a loop

Examples to avoid:

- "This might break in some edge case" without proof
- "Consider adding tests"
- "This could be cleaner"
- "A linter may complain here"
- "This seems inconsistent with the broader architecture" without a concrete defect

## Tooling notes

- Use the available GitHub tooling to inspect pull requests, diffs, files, comments, and to create inline review comments.
- Do not use web fetch for GitHub review tasks when repository tooling is available.
- Do not make exploratory tool calls with no clear purpose.

## Inline comment link format

When you include a repository link in a comment, use this format exactly:

`https://github.com/<owner>/<repo>/blob/<full_sha>/<path>#L<start>-L<end>`

Requirements:
- Use the full commit SHA.
- Use the correct repository name.
- Include the `#L<start>-L<end>` suffix.
- Include at least one line of context before and after the referenced lines when practical.

## Laravel appendix for high-signal reviews

Use this appendix only when the pull request changes Laravel backend PHP code.

Treat this appendix as supporting review guidance, not as a standalone reason to comment. Do not flag a change only because it differs from a Laravel preference or convention. Only report an issue when the changed code introduces a concrete bug, security problem, or meaningful performance problem with high confidence.

### High-signal Laravel findings worth checking

Flag a Laravel-specific issue only when it is clearly introduced by the pull request and one of the following is true:

- The change introduces a clear N+1 query pattern, such as loading related data inside a loop when the relation could have been eagerly loaded.
- The change uses raw SQL with user-controlled input, or otherwise constructs a query in a way that clearly bypasses safe parameter binding.
- The change introduces unsafe mass assignment or broad unvalidated writes, such as using `$request->all()` or equivalent unvalidated input in a create or update path.
- The change removes or bypasses authorization for a sensitive action that should clearly be protected.
- The change introduces a file upload path without essential validation of type, extension, or size when those checks are clearly required.
- The change adds or modifies an external HTTP call without an explicit timeout or without checking for failure, in a way that can clearly hang, mask errors, or mis-handle bad responses.
- The change introduces queue or job settings that are clearly unsafe, such as a job `timeout` that is greater than or equal to `retry_after`, causing duplicate execution risk.
- The change introduces an obvious performance problem in changed code, such as repeatedly counting related records instead of using an aggregated query, or loading far more rows or columns than needed in a hot path.
- The change introduces a configuration or secret-handling problem that is directly visible in the diff, such as reading `env()` outside config files in application code or storing sensitive values in an unsafe way.

### Laravel-specific things to avoid flagging

Do not flag any of the following unless the pull request creates a concrete defect:

- Preference for one Laravel pattern over another
- Whether validation is inline or in a Form Request, unless the chosen approach causes a real bug or missing validation
- Controller length, service extraction, naming, or architectural style by themselves
- Missing tests
- Pure consistency issues with how the rest of the codebase uses Laravel
- Caching suggestions, query refactors, or Eloquent rewrites that are only theoretical improvements

### Laravel review heuristics

When evaluating Laravel code, prefer comments that are tied to a concrete failure mode:

- Explain what request or execution path breaks.
- Explain whether the problem affects correctness, security, or performance.
- Point to the exact changed line or block that introduces the issue.
- If the issue depends on assumptions outside the diff and you cannot validate them, do not comment.

### Laravel reference areas

If you need supporting context for a Laravel-specific concern, focus on these areas from the Laravel best practices skill rather than treating the whole skill as mandatory review criteria: database performance and N+1 avoidance, security and authorization, validation and safe input handling, queue and job safety, routing and controller request handling, and HTTP client timeouts and error handling.
